# GCFeed 简历亮点

本文只整理当前项目已经实现的能力，用于简历项目描述和面试讲解。

## 一句话描述

GCFeed 是一个面向短视频场景的高性能 Feed 流后端系统，围绕时间线 Feed、关注流、推荐流、热榜流、视频分发、互动计数、曝光反馈和播放优化构建完整内容消费链路。

## 简历项目亮点

### Feed 流核心链路

1. **稳定游标分页**
   时间线 Feed、关注流、热榜流、推荐流、评论列表和消息列表均支持游标分页，基于发布时间、视频 ID、热度分、排序分等字段生成下一页游标，提升翻页稳定性。

2. **Feed 页缓存**
   Feed 页缓存只保存视频 ID、作者 ID、发布时间、热度分等轻量字段，降低列表查询、排序和缓存存储成本。

3. **ID-Detail 分离查询**
   Feed 先获取稳定排序的视频 ID 列表，再批量聚合视频详情、作者信息和互动计数，减少 N+1 查询并提升列表读取性能。

4. **批量详情聚合**
   Feed 组装阶段批量读取视频卡片和互动计数，并按原 Feed 排序结果恢复返回顺序。

5. **视频卡片缓存**
   视频标题、封面、媒体地址、作者昵称、作者头像等相对稳定字段独立缓存，降低 Feed 卡片组装回源成本。

6. **互动计数缓存**
   点赞数、评论数、收藏数独立缓存，降低高频互动计数读取压力。

7. **缓存降级**
   Feed 页、视频卡片、互动计数缓存读取异常时继续回源数据库，保障 Feed 主链路可用。

8. **singleflight 合并回源**
   Feed 页缓存未命中时合并同 Key 回源请求，降低缓存击穿带来的数据库瞬时压力。

9. **TTL 抖动**
   Feed 页缓存 TTL 加入抖动，降低缓存集中失效风险。

### 关注流与视频分发

1. **推拉结合关注流**
   小作者发布走粉丝 Inbox 推送，大 V 作者走 Author Outbox 拉取合并，控制发布扩散成本。

2. **关注流分批 Fanout**
   视频发布后按粉丝 ID 游标分批写入关注流，避免单次 fanout 写入过大。

3. **关注关系回填**
   用户新关注作者后，回填作者近期内容到关注流，提升关注后的内容可见性。

4. **Feed 预热**
   视频发布事件触发视频卡片和互动计数缓存预热，降低新内容首次进入 Feed 时的回源成本。

5. **视频发布事件驱动**
   视频发布后投递发布事件，联动 Feed 分发、缓存预热和视频向量生成。

### 异步解耦与互动性能

1. **异步互动落库**
   点赞、收藏先写 Redis 状态和计数，再投递消息给 Worker 异步写入数据库，提升接口响应速度。

2. **异步事件拆分**
   点赞收藏、视频发布、曝光观看、视频向量生成等链路通过消息队列拆分，降低接口链路耦合。

3. **RabbitMQ 持久化消息**
   异步事件使用持久化消息，消费端显式 ACK/NACK，失败任务支持重新入队。

4. **写接口幂等**
   视频发布、点赞收藏、评论、关注、消息、播放 QoS 等写操作支持幂等键，保障客户端重试结果稳定。

5. **重复消费保护**
   Worker 消费端依赖唯一键和幂等键，避免重复消息导致业务事实重复写入。

6. **事务内计数更新**
   点赞、收藏、评论等事实写入与统计计数更新放在同一事务，保障持久化计数稳定。

7. **互动计数最终一致**
   点赞、收藏高频计数先走缓存状态，后台异步落库到统计表，缓存短 TTL 回源修正。

8. **热点计数分片**
   点赞、收藏计数增量使用分片 Key，分散热点视频的计数写压力。

### 热榜与推荐系统

1. **热榜统计**
   基于点赞、收藏、评论权重计算热度分，支持热门内容排序。

2. **分钟桶热榜**
   使用 Redis ZSET 维护分钟级热度桶，支撑短时间窗口内的热门内容统计。

3. **滑动窗口热榜**
   读取热榜时合并最近 60 个分钟桶，形成一小时滑动热榜。

4. **推荐候选召回**
   推荐系统从候选池读取公开视频，并在候选池阶段过滤近期已曝光内容。

5. **推荐排序打分**
   推荐排序融合相似度、热度分和新鲜度分，生成推荐流排序结果。

6. **作者打散**
   推荐结果按作者做打散，提升内容来源多样性。

7. **曝光去重**
   推荐决策阶段基于用户近期曝光记录过滤重复内容。

8. **曝光聚合索引**
   维护用户-视频曝光聚合表，记录首次曝光、最近曝光、曝光次数和最近曝光场景。

9. **观看行为采集**
   记录曝光、播放、进度、完播等观看事件，为推荐反馈和用户画像提供数据。

10. **用户兴趣向量**
    基于播放、完播、观看时长等正反馈事件聚合用户兴趣向量。

11. **视频向量生成**
    视频发布后异步生成视频向量，供推荐相似度计算使用。

### 播放与上传优化

1. **播放配置兜底**
   根据平台和网络类型读取播放配置，配置缺失时回落到默认配置。

2. **视频预加载建议**
   提供当前视频后的预加载视频列表，优化短视频连续播放体验。

3. **播放 QoS 上报**
   支持首帧耗时、卡顿次数、观看时长等播放质量数据上报，并支持幂等写入。

4. **媒体 Range 支持**
   静态视频资源支持 Range 请求，适配浏览器拖动进度和分段加载。

5. **上传链路校验**
   上传接口限制文件大小、扩展名、MIME、视频时长、分辨率和编码格式。

6. **MP4 faststart**
   上传 MP4 后做 faststart 处理，优化视频在线播放首帧体验。

### 账号、消息与安全

1. **可选登录态 Feed**
   Feed 接口支持可选 JWT，兼容游客访问和登录用户个性化场景。

2. **JWT 登录态**
   支持用户注册、登录、资料读取和更新，并通过 JWT 维护访问身份。

3. **内部接口鉴权**
   推荐、曝光等内部接口通过 X-Internal-Token 做服务间鉴权。

4. **消息中心**
   点赞、评论、关注等事件生成站内消息，并支持消息分页、未读数和已读标记。

5. **消息幂等写入**
   通过事件 ID 或幂等键避免重复生成站内消息。

### 工程化与部署

1. **分层架构**
   后端按 Domain、Application、Infrastructure、Interfaces 分层，业务逻辑、接口层和基础设施实现清晰拆分。

2. **依赖注入**
   Redis、RabbitMQ、JWT、Repository 等能力通过接口注入，便于测试和替换实现。

3. **自动迁移与索引**
   启动时自动迁移数据模型，并创建 Feed 时间线索引，优化回源查询。

4. **Docker Compose 本地编排**
   支持 API、Worker、Web、MySQL、Redis、RabbitMQ 一键本地启动。

5. **Kubernetes 部署**
   提供 API、Worker、Web、MySQL、Redis、RabbitMQ 的部署清单、Service、PVC 和健康检查。

6. **测试覆盖**
   覆盖 Feed、推荐、曝光、互动、播放、上传、消息、关系、视频发布 Fanout 等核心链路。

## 简历压缩版

### 版本一

GCFeed 是一个面向短视频场景的高性能 Feed 流后端系统，支持时间线 Feed、关注流、推荐流、热榜流、视频分发、点赞评论、消息通知、曝光反馈与播放优化；围绕稳定游标分页、Feed 缓存、ID-Detail 分离查询、推拉结合、异步解耦、热榜统计、曝光去重、推荐排序和 Kubernetes 部署优化 Feed 读取性能与高频互动吞吐。

### 版本二

围绕短视频 Feed 核心链路，设计稳定游标分页、Feed 页缓存、视频卡片缓存、互动计数缓存、ID-Detail 分离查询、推拉结合关注流、异步互动落库、热榜分钟桶、滑动窗口热榜、曝光去重、用户兴趣向量和播放预加载，提升内容分发效率、列表读取性能和连续播放体验。

### 版本三

负责 GCFeed 短视频 Feed 流后端系统设计与实现，完成时间线 Feed、关注流、推荐流、热榜流、视频分发、互动计数、曝光反馈、消息中心和播放优化等核心能力，并通过缓存降级、singleflight、TTL 抖动、批量聚合、异步解耦和推拉结合提升系统性能与稳定性。

## 面试展开顺序

1. 先讲 Feed 读取链路：稳定分页、页缓存、ID-Detail 分离、批量详情聚合。
2. 再讲分发链路：视频发布事件、推拉结合关注流、分批 fanout、Feed 预热。
3. 再讲互动链路：点赞收藏缓存、异步落库、幂等、计数最终一致。
4. 再讲推荐链路：候选召回、曝光去重、向量相似度、热度新鲜度融合、作者打散。
5. 最后讲工程化：RabbitMQ ACK/NACK、Kubernetes 部署、测试覆盖。
